## 📝 重要说明

### 关于两个仓库

你的项目使用了**双仓库模式**：

1. **源码仓库**（当前）
   - 仓库：Parade001/Parade001.github.io
   - 分支：master
   - 内容：Markdown 文章、配置文件、主题源码
   - 用途：版本控制和源码管理

2. **部署仓库**（自动推送）
   - 仓库：Parade001/Parade001.github.io
   - 分支：master（由 hexo deploy 推送）
   - 内容：HTML、CSS、JS 等静态文件
   - 用途：GitHub Pages 托管

**注意**：根据你的 _config.yml 配置，源码和部署使用的是**同一个仓库**。这意味着：
- 源码在 master 分支
- 部署产物也会推送到 master 分支
- 这可能会导致冲突

### 🔧 建议的改进方案

**方案 1：使用不同分支（推荐）**
```yaml
# _config.yml
deploy:
  type: git
  repo: https://github.com/Parade001/Parade001.github.io.git
  branch: gh-pages  # 改为 gh-pages 分支
```

然后在 GitHub 仓库设置中，将 Pages 源设置为 `gh-pages` 分支。

**方案 2：使用两个独立仓库**
- 源码仓库：hexoblog（私有）
- 部署仓库：Parade001.github.io（公开）

### ⚠️ 当前配置的风险

由于源码和部署都在同一个仓库的 master 分支，可能会出现：
1. 源码和构建产物混在一起
2. 每次部署都会覆盖源码提交
3. 难以回滚到特定版本
